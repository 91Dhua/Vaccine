## 1. 概述

本文说明在**母猪—公猪匹配**过程中精液**资质过滤**如何工作，以及**近交**应如何评估。

面向实现备选公猪推导与排序的**后端工程师**。

目的在于区分：

1. 决定某公猪对该母猪是否**有资格**考虑的规则；与  
2. 用于估计某配对是否带来不可接受近交风险的计算。

本文聚焦**匹配资质**。预留分配发生在公猪已被选定之后。

## 2. 资质过滤在流程中的位置

### 2.1 匹配先于预留

匹配流程顺序为：

1. 检测到母猪发情；
2. 判断母猪是否有手动指定或规划意图；
3. 对候选公猪应用资质过滤；
4. 对仍合格的公猪排序；
5. 选定公猪；
6. 再针对该公猪尝试对**未预留**可用库存做预留分配。

为便于计算，后端也可在批次中**第一头**母猪发情时对整批做一次演算。

该模型下：

1. 首次发情事件时，可一并处理当前批内群体；
2. 可为范围内母猪推导备选公猪列表与排序；
3. 若后续有新母猪入批，在其发情时再单独匹配。

### 2.2 为何要与预留分离

过滤回答的是：

「这头公猪**能否**作为该母猪的候选？」

并不回答：

「系统能否为该所选公猪分配足够的未预留可用精液库存？」

后者在匹配选定公猪之后解决。

## 3. 资质过滤顺序

### 3.1 分层顺序

已文档化的匹配流程按以下分层顺序应用过滤：

1. 手动母猪—公猪指定；
2. 规划意图或品种意图解析；
3. 资质过滤；
4. 排序；
5. 容量与预留分配检查。

### 3.2 规划意图的影响

规划意图主要在常规资质过滤与排序**之前**，增加一层品种或精液方向过滤。

若存在规划意图，会先收窄候选池：

1. `肉猪生产` → 终端系精液候选池；
2. `种猪扩繁` → 母猪系精液候选池；
3. `混合` → 按配置比例将母猪划入上述两向之一；
4. `手动品种分配` → 候选池收窄为用户定义的目标品种。

若无规划意图，则跳过该品种方向过滤，继续其余备选逻辑。

### 3.3 后端推导摘要

为单头母猪推导备选公猪列表时，后端逻辑应：

1. 从猪场可用公猪目录出发；
2. 若存在手动指定，优先应用；
3. 若存在配种意图，应用规划意图过滤；
4. 应用可用性过滤，排除标记为 **停产（Off Production）** 的公猪；
5. 应用生产线过滤；
6. 若存在系谱数据，计算 COI；
7. 将剩余公猪送入排序。

以上步骤的输出即为备选公猪候选列表。

## 4. 各过滤项定义

### 4.1 手动指定

若母猪存在手动母猪—公猪指定，则**优先**使用该公猪。

手动指定绕过常规自动候选排序，但依产品规则仍可能做下游策略或容量检查。

### 4.2 品种或精液方向过滤

若存在配种计划，系统须先判定该母猪可接受何种精液：

1. `肉猪生产` → 终端系精液；
2. `种猪扩繁` → 母猪系精液；
3. `混合` → 按配置百分比将母猪划入母猪系或终端向；
4. `手动品种分配` → 母猪须匹配已配置的目标品种之一。

此步尚非近交或健康决策，仅为候选集的第一道收窄。

### 4.3 可用性过滤

公猪在计划配种时间须**在运营上可配种**。

至少包括：

1. 公猪未标记为 **停产（Off Production）**；
2. 公猪仍可对新的配种需求开放。

若未通过此过滤，配对立即拒绝。

健康状况**不**作为此硬过滤的一部分，而进入排序信号。

### 4.4 生产线过滤

公猪所在单元须通过已配置的生产线映射，与母猪所属生产线关联。

确保公猪在母猪配种所处的运营上下文中有效。

若公猪未关联到母猪生产线，配对拒绝。

### 4.5 近交评估（COI）

若存在系谱数据，系统应计算子代预期**近交系数（COI）**。

默认**不**作为硬资质直接拒绝。

而是：

1. `COI ≤ 10%` 视为安全；
2. `COI > 10%` 默认仍可配对；
3. COI 更高者应排在 COI 更低者之后；
4. 凡 `> 10%` 须在界面标为 **高 COI（High COI）** 并展示实际百分比。

### 4.6 健康作为排序信号

健康本身不应单独将公猪从备选列表中剔除。

通过硬资质过滤后，健康进入排序：

1. 健康标签更少的公猪应排在更多者之前；
2. 排序惩罚基于该公猪记录的健康标签数量；
3. 若其他条件相同，更健康者排名更高。

### 4.7 库存作为后续运营闸门

库存不应与上述遗传或运营资质过滤混淆。

概念上：

1. 过滤决定公猪**是否可考虑**；
2. 排序决定**应选哪头**合格公猪；
3. 预留分配检查所选公猪是否能分配足够的未预留可用库存。

部分文档将可用库存描述为实践上的资质闸门；但完整库存是否充足，在选定后的容量与预留步骤才最终落定。

## 5. 近交如何评估

### 5.1 目标

近交检查用于降低子代从父母双方因共同祖先而继承过多相同遗传物质的风险。

系统力求在分配精液**之前**阻断高风险母猪—公猪配对。

### 5.2 近交系数的含义

近交系数常记为 `F`，表示子代从共同祖先处**同源**获得两个等位基因拷贝的概率。

`F` 越高，母猪与公猪遗传亲缘越近。

### 5.3 常用参考值

常见参考点：

1. 堂表亲交配：`F = 6.25%`
2. 同父异母/同母异父：`F = 12.5%`
3. 全同胞：`F = 25%`
4. 亲子：`F = 25%`

本产品后端行为约定：

1. `COI ≤ 10%` 视为安全；
2. `COI > 10%` 视为高 COI；
3. 高 COI 配对默认仍可配对，但必须排序靠后并明确展示实际百分比。

### 5.4 系谱深度要求

为有意义地计算 `F`，母猪与公猪双方系谱应至少覆盖**三代**。

代数越深精度越好；浅系谱往往低估真实近交风险。

### 5.5 计算公式

Wright 近交系数标准路径形式为：

`F_offspring = Σ (1/2)^(n1 + n2 + 1) * (1 + F_A)`

其中：

1. `Σ` 表示对所有有效共同祖先路径求和；
2. `n1` 为公猪侧到共同祖先的世代数；
3. `n2` 为母猪侧到共同祖先的世代数；
4. `F_A` 为共同祖先自身的近交系数。

每个共同祖先对最终 `F` 均有一部分贡献。

### 5.6 解读

在备选路径中，最终 `F` **本身**默认不作为硬拒绝规则。

而是：

1. `≤ 10%` 视为安全；
2. `> 10%` 视为高 COI；
3. 在其余条件相同的合格配对中，优先更低 `F`；
4. 高 COI 须在界面展示实际 COI 百分比。

## 6. 当前系统行为与目标计算

### 6.1 先前文档描述的行为

先前文档将近交规则描述为：

「拒绝共享禁止祖先深度的公猪」

这更接近基于系谱的**排除规则**，而非完整基于系数的 COI 评估。

旧模型下系统行为更接近：

1. 检查母猪与公猪已知祖先；
2. 判定是否在禁止深度内共享祖先；
3. 若是则拒绝配对。

### 6.2 先前规则存在的原因

该简化护栏仍有价值：在尚未具备足够结构化系谱以可靠计算每对配对 `F` 时，仍能阻断明显高风险配对。

### 6.3 后端目标行为

目标状态更精确：

1. 收集母猪与公猪系谱；
2. 识别所有共同祖先；
3. 计算子代近交系数 `F`；
4. `F ≤ 10%` 归类为安全；
5. `F > 10%` 归类为高 COI；
6. 高 COI 默认仍可配对；
7. 按 COI 从低到高排序配对问题程度；
8. 当 `F > 10%` 时，界面展示 **高 COI** 及实际百分比。

### 6.4 缺失系谱

若系谱缺失或不完整：

1. 当前护栏可能需跳过；
2. 系统应降低置信度；
3. 界面应明确提示无法完整评估 COI。

## 7. 近交计算示例

### 7.1 堂表亲式配对

若母猪与公猪共享同一祖父母且路径无额外近交，单路径贡献通常为：

`(1/2)^(2 + 2 + 1) = 1/32 = 3.125%`

若存在两条此类共享祖父母路径，合计：

`2 * 3.125% = 6.25%`

此即堂表亲交配常引用 `F = 6.25%` 的原因。

### 7.2 半同胞配对

若共享一方父母，贡献为：

`(1/2)^(1 + 1 + 1) = 1/8 = 12.5%`

高于本产品 `10%` 安全阈值，默认仍可配对，但应排为 **高 COI**，界面显示 `12.5%`。

### 7.3 全同胞或亲子

二者均可得：

`F = 25%`

属极高 COI，应排在问题最严重一端，并在界面清晰展示实际 COI 百分比。

## 8. 过滤通过之后

### 8.1 后端如何排序备选列表

过滤完成后，剩余公猪进入排序阶段。

后端应按以下顺序排序备选公猪列表：

1. 全疗程库存覆盖（能覆盖母猪完整授精疗程所需剂量的优先）；
2. 健康负担（健康标签数量）；
3. COI 严重程度（从低到高）；
4. 采精冷却就绪程度；
5. FEFO（先到期先用）契合度；
6. 当前批次内负载均衡；
7. 稳定决胜规则。

实现上可表述为：

1. 对完整授精疗程有足够未预留可用库存的公猪，优先于仅部分覆盖者；
2. 其中健康标签更少者优先；
3. 其中 COI 更低者优先；
4. 其中距冷却结束更远/更满足冷却者优先（依产品定义）；
5. 其中可接受库存更符合 FEFO 者优先；
6. 其中同批次当前分配更少者优先；
7. 仍并列则按 `boarId` 升序。

### 8.2 预留分配

排序选定公猪后，预留步骤检查是否存在足够的**未预留**可用精液。

若无法分配库存，系统应提示短缺并询问用户下一步。

用户可选择：

1. **忽略（Ignore）**  
   用户将自行处理，不需要系统建议。  
2. **回退（Fall back）**  
   系统应级联到排序列表中的下一头合格公猪。
