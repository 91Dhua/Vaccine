## 1. 概述

本文档归纳当前精液**库存事件日志**的行为，以及仍在开发中的**手动精液登记**流程所需的补充行为。

## 2. 目的

本文件用于：

1. 汇总已在其他文档中描述的现有库存事件日志行为；
2. 规定各日志类型在界面中的展示方式；
3. 明确手动精液编码登记如何与库存日志联动；
4. 为后续实现工作提供开发期参考。

## 3. 库存日志界面模型

既有文档已说明：库存变动按**批次**记录在事件日志中。

当前最权威的引用为：

1. `docs/semen/Semen_Management_Feature.md`
  章节 `3.5.6 Inventory Event Log`（库存事件日志）

该文档约定：

1. 每次库存变动都会写入事件日志；
2. 用户可在批次详情中查看该批次历史；
3. 所有事件至少包含：
  - `timestamp`（时间戳）
  - `user`（用户）
  - `batch ID`（批次 ID）

### 3.1 当前界面列

库存日志当前界面列为：

1. `Log Type`（日志类型）
2. `Time`（时间）
3. `User`（用户）
4. `Details`（详情）

### 3.2 展示原则

每条库存事件在界面中对应**一行**日志。

该行应始终展示：

1. 事件类型 → `Log Type`；
2. 事件时间 → `Time`；
3. 操作用户，或 `System` → `User`；
4. 简明可读摘要 → `Details`。

## 4. 日志类型与界面展示

### 4.1 核心日志类型

当前已文档化的日志类型包括：

1. `created`（创建/入库）
2. `lab_tested`（实验室检测）
3. `diluted`（稀释）
4. `reserved`（预留）
5. `picked`（拣选/出库）
6. `used`（使用）
7. `moved`（移动）
8. `adjusted`（调整）
9. `discarded`（废弃）
10. `manual_entry`（手动录入）
11. `expired`（过期）

### 4.3 「详情」列格式规则

`Details` 列应：

1. 为人类可读文本，而非原始 JSON；
2. 包含能说明「发生了什么」的关键字段；
3. 优先短语，避免冗长整句；
4. 在相关时包含母猪 ID、预留 ID、拣货单 ID、公猪 ID 等标识。

### 4.4 手动登记在界面中的展示

对非 Sentri 精液编码的登记流程，除非日后引入专门事件类型，否则日志行应展示为 `manual_entry`。

`Details` 至少应包含：

1. 扫描码上下文；
2. 公猪 ID；
3. 品种；
4. 来源场；
5. 到期日。

## 5. 手动精液登记的必要行为

当用户扫描的精液袋编码**不是** Sentri 编码时，系统应先判断该编码是否已登记。

### 5.1 已登记编码

若扫描的非 Sentri 编码已在系统中登记：

1. 无需再次登记；
2. 不应要求用户重复录入相同数据；
3. 系统应直接解析已存储的精液信息。

### 5.2 新编码登记

若扫描的非 Sentri 编码尚未登记：

1. 用户必须提供：
  - 公猪 ID（**必填**）
  - 品种（**必填**）
  - 来源场
  - 到期日
2. 系统应创建精液登记记录；
3. 系统还应为该编码创建一条库存日志记录。

### 5.3 为何库存日志重要

写入库存日志的意义在于：

1. 下次扫描同一编码时，系统应识别为已登记；
2. 用户无需对同一精液编码重复登记；
3. 日志成为可追溯的系统记录，将编码与精液库存历史关联。

## 6. 待明确的实现说明

以下实现细节仍需产品或工程方案进一步明确：

1. 非 Sentri 扫描精液是否**始终**产生 `manual_entry` 事件，或是否应引入专用事件类型；
2. 库存日志条目是在登记当下即创建，还是待更完整的库存记录确认后再创建；
3. 在配种任务流中登记精液（而非独立库存流）时，批次 ID 应如何确定。
