# 免疫效果审核与决策 — 全流程与代码映射（学习笔记）

> 在实现「免疫效果审核」UI 之前，先对齐 **Console ↔ Mobile ↔ 化验闭环** 的交互与数据关系。本文档描述业务阶段、状态机、与当前仓库的对应关系（缺口即后续要做的事）。

---

## 一、端到端阶段（与你描述一致）

### 阶段一：计划制定与下发（Console）

| 能力 | 业务含义 | 当前仓库 |
|------|----------|----------|
| 计划名称、疫苗、剂量、时间框架 | 普免/跟批触发 | `VaccinePlanPage` 等计划配置；任务由 `VaccineTaskWizard` + `App` 写入 `tasks` |
| 目标圈选（高级设置/规则） | 静态/结构特征圈群 | 计划中已有目标猪群、豁免等 UI；与任务 `pigIds`、`targetPigGroupLabel` 应对齐 |
| 豁免规则 | 动态状态不可打 | Mobile 端 `exemptionHit` / `exceptionPending`；计划侧豁免配置需与任务下发同源 |
| **效果追踪** | 开启采样、间隔（如 28 天）、抽样比例、**合格率阈值（如 80%）** | **尚未**与计划实体绑定存储；也**未**触发「采样任务」数据 |

### 阶段二：现场执行（Mobile）

| 能力 | 业务含义 | 当前仓库 |
|------|----------|----------|
| 栋舍-房间-栏位聚合 | 选址 + 任务列表 | `MobileVaccinationPage` + `mobileWorkshops` / `pigMeta` |
| 双检、暂缓、补打 | 豁免/强制/暂缓 → Is_Retry | `mobileVaccinationUtils`、`spawnRetryTask`、补打清单、本间完成 |
| 与 Console 同源 | 共享 `mobilePigTasks` | `App.tsx` 状态提升 |

**缺口**：**抽血采样任务**（阶段三）在 Mobile 侧 **未**建模，不能与「免疫任务」混为同一列表 unless 产品定义为统一「现场任务」聚合。

### 阶段三：效果抽检与化验（Mobile + Console）

| 步骤 | 业务含义 | 当前仓库 |
|------|----------|----------|
| 采样触发 | 达间隔后下发 Mobile 采样任务 + 随机名单 | **无** |
| 更换样本猪 | 淘汰/死亡替换 | **无** |
| 结果录入 | 实验室结果在 **Console** 录入 | **无**（上次误加的抽屉已回退） |

### 阶段四：智能决策与闭环（Console）

| 分支 | 条件 | 动作 |
|------|------|------|
| PASS | 合格率 ≥ 阈值 | 原始计划/批次 **归档闭环**（状态写入、不可再下发冲突任务） |
| FAIL | 合格率 < 阈值 | 红色预警；**必须**勾选「排除野毒」；按 **实时存栏**（扣死淘）克隆原疫苗/剂量生成 **【效果补免】** 任务 → 回到阶段二 |

**缺口**：合格率计算依赖 **实录样本数、阳性数、阈值**；补免依赖 **关联免疫任务 ID、原疫苗、剂量、存栏接口**。

## 二、「免疫效果审核」页面应嵌入的交互顺序（建议）

1. **入口**：从「待化验批次列表」进入（每行绑定：`计划/批次`、`采样时间`、`关联任务 ID`、`目标抽样数`、`阈值`）。不是孤立抽屉，而是 **有上下文的一行 → 再打开抽屉/全屏表单**。
2. **只读头**：展示溯源（标题 + 状态 Badge + 采样/批次/任务 ID），与列表行一致。
3. **录入**：目标抽样数（只读，来自计划）；实录样本数；抗体阳性合格数；阈值只读。
4. **实时合格率**：`阳性 / 实录样本数`（实录为 0 时禁用或无意义，需防除零）。
5. **决策**  
   - **≥ 阈值**：绿色结论 + 「确认并结案」→ 写回批次状态 + 计划归档。  
   - **< 阈值**：红色预警区 + Checkbox（未勾选则「生成补打任务」禁用）+ 补免预览（疫苗/剂量来自 **关联任务**；存栏来自 **实时接口**）+ 期望下发时间 + 「生成补打任务」→ 创建带 **效果补免** 标签的 `TaskRow` / 下发 Mobile。

---

## 三、与现有数据结构的衔接建议（后续编码时）

- **关联免疫任务**：使用现有 `TaskRow.id` 或未来 `VAC-xxx` 与之一一映射。
- **补免任务**：在 `TaskRow` 上增加可选字段，如 `taskKind: 'effect_booster' | 'normal'`、`sourceTaskId`、`sourcePlanId`，便于 Mobile 与报表筛选。
- **不要在未接 Tailwind 栈前**强行混用两套样式；Console 页优先 **Ant Design + `styles.css`**，与 `VaccineTaskListPage` 一致。

---

## 四、上次实现被回退的原因（备忘）

- 未先绑定 **列表 → 单条审核上下文**，抽屉缺少与任务/计划的数据流。
- 引入 **Tailwind** 与全局构建变更，与现有 **纯 Ant + 自定义 CSS** 栈不一致，增加维护成本。
- 「效果追踪」在计划侧尚未落地，审核页缺少 **触发来源**（哪次采样、哪条计划）。
